# 视图切换设计备忘

## 当前目标

为主界面文件列表增加：

- 列表视图
- 分组视图

分组视图中的组行仅显示实际添加路径，例如：

- `Y:\RUANSTRM\动漫刮削好的\无职转生～到了异世界就拿出真本事～ [tmdbid=94664]\Season 2`
- `Y:\RUANSTRM\动漫刮削好的\无职转生～到了异世界就拿出真本事～ [tmdbid=94664]\Season 1`

## 方案 A：保守落地方案

适用目标：

- 尽量少改现有识别、重命名、归档、刮削流程
- 在现有 `ttk.Treeview` 基础上增加列表/分组切换
- 通过下方详情面板解决长路径、长状态、长目标路径的完整显示问题

核心做法：

- 保留 `self.file_list` 作为唯一业务数据源
- `Treeview` 支持列表视图和分组视图两种渲染
- 分组视图按实际添加路径分组
- 组节点只显示添加路径
- 新增详情面板，选中组或文件时自动换行显示完整内容

已知限制：

- 原生 `ttk.Treeview` 不支持真正的单元格自动换行
- 行高不能随内容自动增长
- 因此不能保证所有长文本都在表格单元格内部“完美自动换行”

## 方案 B：备用重构方案

触发条件：

- 如果方案 A 的详情面板方案仍然不能满足阅读体验
- 如果必须实现“窗口大小变化时，表格内容本身自动换行、行高自动撑开、无需手动拖列宽”

核心思路：

- 放弃主显示区使用原生 `ttk.Treeview` 作为核心表格
- 改为 `Canvas + Frame + 自定义行组件` 的方式重做列表区
- 每一行使用可换行的 `Label`/文本组件显示内容
- 行高根据内容自动增长
- 分组行和文件行都由自定义组件渲染

方案 B 预期能力：

- 长路径可随窗口宽度自动换行
- 长状态文本可完整显示
- 长归档路径可完整显示
- 可做出更接近网页列表的交互和视觉层级

方案 B 需要重做的部分：

- 列表渲染
- 滚动同步
- 选择逻辑
- 多选逻辑
- 右键菜单命中逻辑
- 分组展开/折叠
- 行状态刷新映射
- 大量数据下的性能优化

方案 B 的主要风险：

- 改动范围明显大于方案 A
- 回归风险更高
- 文件数量很多时可能出现滚动和刷新性能问题
- 现有依赖 `Treeview` 行 id 的逻辑需要全面改造

## 当前决策

- 默认先做方案 A
- 如果方案 A 的显示效果用户不满意，再切换到方案 B
- 方案 B 视为明确保留的后备路线，不删除
